# 프로젝트 작업 헌법

## 목적

이 문서는 이 저장소에서 기능을 설계하고 구현하고 검증할 때 따를 공통 원칙을 정한다. 특정 기능이나 화면에 묶인 규칙이 아니라, 백엔드, 웹, 모바일, 문서 작업 전체에 적용되는 기준이다.

핵심 목적은 단순하다.

- 토큰을 많이 쓰는 것 자체를 품질로 착각하지 않는다.
- 기능은 작게 나누어 실제 동작과 검증으로 판단한다.
- 설계, 계획, 구현, 테스트는 사용자의 목표에 직접 기여해야 한다.
- 장황한 문서보다 명확한 결정과 검증 가능한 결과를 우선한다.

## 기본 원칙

### 1. 토큰 사용량은 품질 보증이 아니다

토큰을 많이 썼다는 사실은 더 많은 문맥을 읽고 더 많은 설명을 만들었다는 뜻에 가깝다. 좋은 결과물은 토큰 사용량이 아니라 다음 기준으로 판단한다.

- 사용자가 필요한 흐름을 끝까지 완료할 수 있는가
- 기존 기능을 깨뜨리지 않았는가
- 권한, 중복 요청, 실패 상황이 처리되는가
- 핵심 동작을 검증하는 테스트가 있는가
- 화면과 API가 실제 사용 방식에 맞게 단순하고 명확한가
- 유지보수할 사람이 이해할 수 있는 구조인가

### 2. 큰 작업은 반드시 작게 자른다

여러 앱, 서버, 데이터 모델, 인증, 테스트를 한 번에 모두 다루면 작업 비용과 실패 가능성이 커진다. 작업은 가능한 한 독립적으로 검증 가능한 단위로 나눈다.

예시는 다음과 같다.

- 백엔드 API
- 프론트엔드 화면
- 모바일 앱 연결
- 데이터 모델 변경
- 테스트 보강
- 문서 정리
- 배포 또는 운영 설정

각 단위는 무엇을 바꾸고, 어떻게 검증하며, 다음 작업과 어떤 계약으로 이어지는지 분명해야 한다.

### 3. 기존 구조를 먼저 따른다

새로운 추상화나 도구를 추가하기 전에 기존 코드의 패턴을 우선한다.

- 기존 디렉터리 구조와 이름 규칙을 따른다.
- 기존 상태 관리, API 호출, 예외 처리, 테스트 스타일을 따른다.
- 같은 문제를 이미 해결한 코드가 있으면 먼저 재사용한다.
- 불필요한 리팩터링은 하지 않는다.

다만 현재 작업을 안전하게 완성하는 데 방해가 되는 구조는 최소 범위로 정리할 수 있다. 이 경우 정리의 목적과 검증 방법을 명확히 한다.

### 4. 사용자 결과를 기준으로 판단한다

좋은 기능은 내부 구현량이 아니라 사용자가 얻는 결과로 판단한다.

- 사용자가 해야 할 일을 더 쉽게 끝낼 수 있는가
- 오류가 이해 가능한 말로 드러나는가
- 같은 행동을 반복해도 데이터가 망가지지 않는가
- 실패했을 때 복구할 수 있는가
- 새 기능이 기존 사용 흐름과 자연스럽게 이어지는가

## 문서 원칙

### 1. 설계 문서는 짧고 결정 중심이어야 한다

설계 문서는 구현 전체를 대신 쓰는 문서가 아니다. 기본값은 짧은 결정 기록이다.

포함할 내용은 다음 정도로 제한한다.

- 이번 범위
- 제외 범위
- 주요 계약
- 데이터 흐름 또는 상태 변화
- 에러 처리
- 검증 기준

코드 전문은 설계 문서에 넣지 않는다. 꼭 필요한 인터페이스, 필드명, 상태값, 명령어만 적는다.

### 2. 계획 문서는 실행 체크리스트여야 한다

계획 문서는 작업자를 압도하는 장문 지시서가 아니라 실행 순서를 잡는 체크리스트여야 한다.

- 파일별 변경 목적을 적는다.
- 실패해야 하는 테스트와 통과해야 하는 테스트를 구분한다.
- 커밋 단위가 필요하면 작게 나눈다.
- 긴 코드 블록은 피한다.
- 기존 패턴을 따라야 하는 위치만 명확히 지정한다.

계획이 구현보다 길어지기 시작하면 범위가 크거나 문서가 과한 것이다. 둘 중 하나를 줄인다.

### 3. 장문 문서는 예외로 둔다

다음 경우에만 긴 설계나 계획 문서를 허용한다.

- 여러 앱과 서버 계약이 동시에 바뀐다.
- 데이터 모델 또는 권한 모델이 크게 바뀐다.
- 되돌리기 어려운 마이그레이션이 포함된다.
- 여러 사람이 같은 계획을 보고 병렬로 작업해야 한다.
- 장애 대응, 보안, 운영 절차처럼 누락 비용이 큰 작업이다.

그 외에는 짧은 결정 기록과 체크리스트로 충분하다.

## 구현 원칙

### 1. 계약을 먼저 고정한다

구현 전에 외부와 맞닿는 계약을 먼저 확인한다.

- API 요청과 응답
- 화면에서 필요한 상태
- 저장되는 데이터
- 권한 경계
- 실패 시 응답
- 중복 요청 처리

계약이 불명확하면 구현량이 늘고 테스트가 흔들린다.

### 2. 한 번에 한 경계를 넘는다

작업은 가능한 한 하나의 경계를 중심으로 진행한다.

- 서버 내부 로직
- 서버와 클라이언트의 API 계약
- 클라이언트 화면 상태
- 저장소 스키마
- 인증과 권한

여러 경계를 동시에 넘는 작업은 작은 단계로 나누고, 각 단계마다 검증한다.

### 3. 새 추상화는 필요할 때만 만든다

추상화는 코드가 길어 보인다는 이유만으로 만들지 않는다. 다음 중 하나가 분명할 때만 추가한다.

- 의미 있는 중복을 줄인다.
- 테스트할 단위를 분리한다.
- 변경 가능성이 높은 부분을 격리한다.
- 기존 프로젝트 패턴과 자연스럽게 맞는다.

그 외에는 명확한 로컬 구현이 낫다.

### 4. 변경 범위 밖의 코드는 건드리지 않는다

관련 없는 포맷팅, 이름 변경, 구조 변경은 하지 않는다. 작업 중 발견한 개선점은 현재 목표에 직접 필요하지 않으면 별도 메모나 후속 작업으로 남긴다.

## 테스트와 검증

### 1. 테스트는 위험 기준으로 선택한다

모든 것을 테스트하려고 하다가 작업이 과해지면 안 된다. 대신 깨지면 큰 문제가 되는 지점을 우선 검증한다.

- 권한 차이
- 상태 변화
- 중복 요청
- 데이터 생성 또는 삭제
- 외부 서비스 실패
- 네트워크 실패
- 잘못된 입력
- 기존 핵심 흐름의 회귀

### 2. 완료 선언은 검증 뒤에만 한다

작업 완료는 구현이 끝났다는 뜻이 아니라 검증이 끝났다는 뜻이다.

완료 보고에는 다음을 포함한다.

- 변경한 주요 파일
- 실행한 검증 명령
- 통과한 테스트
- 실행하지 못한 검증과 그 이유
- 남은 위험

검증하지 않은 것은 검증했다고 말하지 않는다.

### 3. 실패는 먼저 재현한다

버그 수정은 가능한 한 실패를 먼저 재현한다.

- 실패 조건을 확인한다.
- 최소 재현 테스트나 명령을 만든다.
- 수정 후 같은 경로로 통과를 확인한다.

재현이 어려운 문제는 관찰한 증상, 추정 원인, 확인하지 못한 부분을 구분해서 기록한다.

## 토큰 절약 규칙

### 1. 기본은 저토큰 실용 모드

명시적으로 상세 설계가 필요한 경우가 아니면 다음 방식을 기본으로 한다.

- 긴 배경 설명을 줄인다.
- 코드 전문이 들어간 계획 문서를 만들지 않는다.
- 먼저 변경 범위를 작게 잡는다.
- 필요한 파일만 읽는다.
- 구현 후 핵심 검증 결과를 짧게 보고한다.

### 2. 반복 설명을 줄인다

이미 확정된 내용은 매번 다시 길게 설명하지 않는다. 이후 작업에서는 짧은 이름으로 참조한다.

예시는 다음과 같다.

- "백엔드 API만 진행"
- "화면 연결만 진행"
- "테스트 보강만 진행"
- "문서 정리만 진행"

필요한 파일을 읽고, 필요한 만큼만 설명하고, 검증 결과를 중심으로 보고한다.

### 3. 토큰을 쓸 때는 이유가 있어야 한다

토큰을 많이 써도 되는 경우는 있다. 단, 이유가 분명해야 한다.

- 기존 구조가 복잡해서 맥락 확인이 필요하다.
- 보안, 권한, 데이터 손실 위험이 있다.
- 여러 플랫폼의 계약을 맞춰야 한다.
- 사용자가 장문 설계나 상세 설명을 요청했다.
- 실패 원인을 추적하기 위해 로그와 테스트를 비교해야 한다.

이유 없이 길어지는 설명은 줄인다.

## 의사결정 기준

기능 추가나 작업 범위가 애매할 때는 다음 순서로 판단한다.

1. 지금 사용자가 막힌 흐름을 직접 해결하는가
2. 기존 구조와 자연스럽게 맞는가
3. 테스트 가능한 작은 단위로 구현 가능한가
4. 다음 작업의 기반이 되는가
5. 유지보수 비용이 과하지 않은가

이 기준을 통과하지 못하면 이번 범위에서 제외하고 별도 작업으로 남긴다.

## 작업 방식 약속

앞으로 이 프로젝트에서는 다음을 기본값으로 삼는다.

- 기능은 작게 자른다.
- 문서는 짧게 쓴다.
- 코드보다 먼저 계약과 권한을 확인한다.
- 토큰 사용량이 아니라 검증 결과로 품질을 판단한다.
- 여러 플랫폼을 한 번에 묶지 않고 필요한 순서대로 연결한다.
- 사용자가 원하지 않는 장문 계획은 만들지 않는다.
- 완료 보고는 변경 내용보다 검증 결과를 더 중시한다.

이 헌법의 목적은 작업을 덜 하는 것이 아니다. 불필요한 설명과 과한 계획을 줄이고, 실제로 동작하는 제품을 더 안정적으로 만드는 것이다.
